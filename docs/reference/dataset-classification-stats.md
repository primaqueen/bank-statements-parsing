Результат валидации на всех top-level text-like файлах:

- mismatch после trim trailing `NUL`: `0`

Критичное уточнение:

- без trim trailing `NUL` было `2` ложных mismatch на очень больших файлах
- причина: после `КонецФайла` шли десятки мегабайт `\\x00`

Практический fast path:

1. Читать только начало файла.
2. Читать логический хвост файла, предварительно отбросив trailing `\\x00`.
3. Проверять header `1CClientBankExchange` в первых строках.
4. Искать токены по bytes, без полного декодирования файла.

## How To Distinguish Alternate Encoding From Invalid File

Ниже рабочий алгоритм, который должен отличать:

- `это 1C в другой кодировке`
- `это не 1C или структурно неправильный файл`

### Step 0: Reject obvious non-text containers first

До любой попытки декодирования нужно проверить сигнатуры:

- `PK\\x03\\x04` -> ZIP container
- `Rar!\\x1a\\x07` -> RAR container
- `7z\\xbc\\xaf\\x27\\x1c` -> 7z container

Если сигнатура контейнера найдена, это не “другая кодировка текста”, а другой тип файла.

### Step 1: Find header with a narrow fallback

Нужно проверить первые 1-2 строки:

- строка 1 == `1CClientBankExchange`
- или строка 2 == `1CClientBankExchange` после пропуска служебной строки вроде `Formatvalue`

Если header не найден в первых двух строках:

- файл не нужно считать `1C` только потому, что эта строка где-то встречается дальше
- это `not_1c` или иной формат

### Step 2: Detect token family by bytes, not by generic charset detector

После header classifier должен искать ключевые токены по трем семействам bytes:

- `cp1251`
- `UTF-8`
- `cp866`

Минимальный набор токенов для проверки семейства:

- `ВерсияФормата=`
- `Кодировка=`
- `РасчСчет=`
- `СекцияРасчСчет`
- `КонецФайла`

Если файл не дает ни одного согласованного семейства токенов, это не “альтернативная кодировка”, а структурно проблемный или чужой файл.

### Step 3: Require structural coherence

Файл нужно считать `1C в альтернативной кодировке`, если одновременно выполнены условия:

1. найден header `1CClientBankExchange`
2. найдено согласованное семейство токенов (`cp1251` / `UTF-8` / `cp866`)
3. по этому же семейству читаются ключевые структурные маркеры
4. значения основных полей выглядят синтаксически корректно
   - `ВерсияФормата=...`
   - `Кодировка=...`
   - `Дата...=dd.mm.yyyy`
   - `РасчСчет=` с цифровым значением

Если это выполнено, файл нужно трактовать как валидный `1C`, даже если кодировка не `cp1251`.

### Step 4: Use `Кодировка=` as a hint, not as the only source of truth

Поле `Кодировка=` полезно, но не должно быть единственным критерием.

Нормальное соответствие:

- `Windows` / `windows-1251` -> `cp1251`-семейство
- `UTF-8` -> `UTF-8`
- `DOS` / `dos` -> `cp866`

Но если:

- `Кодировка=` повреждено
- или не совпадает с реально найденным семейством токенов

то приоритет должен быть у структурной когерентности токенов и полей, а не у одного заголовочного значения.

### Step 5: When to classify as invalid or broken

Файл нужно считать не “другой кодировкой”, а неправильным / broken, если выполняется хотя бы одно из условий:

1. header не найден в первых двух строках
2. нет ни одного согласованного семейства токенов
3. `Кодировка=` есть, но следующее содержимое распадается и не дает читаемой структуры
4. вместо структуры `1C` виден обычный текст, табличная выгрузка, письмо или бинарный контейнер
5. присутствуют отдельные 1C-фрагменты, но нет базовой формы выписки или экспорта документа

### Practical Rule

Для production classifier правило должно быть таким:

- `encoding_variant`:
  header найден, токены одного из известных семейств согласованы, структура читается
- `broken_1c`:
  header найден, но не удается получить согласованную структуру ни по одному семейству
- `not_1c`:
  header не найден, либо файл является контейнером / письмом / таблицей / другим форматом

### How To Distinguish Alternate Encoding From Invalid File

Нужно считать файл `1C в другой кодировке`, а не `неправильным`, если одновременно выполняются условия:

1. `1CClientBankExchange` найден в первой или второй строке.
2. Найдено согласованное семейство токенов:
   - `cp1251`
   - `utf-8`
   - `cp866`
3. В этом же семействе читаются структурные маркеры:
   - `ВерсияФормата=`
   - `Кодировка=`
   - `РасчСчет=`
   - `СекцияРасчСчет`
   - `КонецФайла`
4. Значения ключевых полей выглядят синтаксически правдоподобно:
   - версия формата похожа на `1.02` / `1.03`
   - `Кодировка=` даёт одно из ожидаемых значений или его очевидный вариант
   - даты похожи на `дд.мм.гггг`
   - счет похож на цифровую строку

Если это выполнено, файл нужно считать валидным `1C`, даже если его кодировка не канонический `Windows`.

Нужно считать файл `broken_1c` или `not_1c`, если:

1. header не найден в первых двух строках;
2. ни одно семейство токенов не даёт согласованной структуры;
3. `Кодировка=` найдено, но следующие структурные токены не читаются в том же семействе;
4. вместо структуры `1C` виден контейнер, письмо, табличная выгрузка или иной формат.

### Important Rule

Поле `Кодировка=` должно использоваться как сильная подсказка, но не как единственный источник истины.

Приоритет проверки такой:

1. header
2. согласованное семейство токенов
3. структурная когерентность
4. значение `Кодировка=...`

То есть:

- `Кодировка=Windows` при читаемых `cp1251` токенах -> нормальный `1C`
- `Кодировка=DOS` при читаемых `cp866` токенах -> нормальный `1C`
- `Кодировка=UTF-8` при читаемых `utf-8` токенах -> неканоничный, но валидный для нашего classifier
- поврежденное `Кодировка=` при читаемой остальной структуре -> `1C` с поврежденным header field, а не автоматически `not_1c`

## Practical Takeaways

### 1. Как быстрее всего оценивать очень большие файлы

Рабочий алгоритм:

1. Прочитать только первые `32 KB`.
2. Найти логический конец файла, отбросив trailing `\\x00`.
3. Прочитать только последние `8 KB` логического хвоста.
4. Проверить header в первых строках.
5. Искать ключевые токены по bytes:
   - `ВерсияФормата=`
   - `Кодировка=`
   - `РасчСчет=`
   - `СекцияРасчСчет`
   - `СекцияДокумент=`
   - `КонецДокумента`
   - `КонецФайла`
6. Учитывать оба варианта token encoding:
   - `cp1251`
   - `UTF-8`
   - `cp866`

### 2. Как оценивать архивы

Для текущей задачи достаточно estimate-подхода:

1. Открывать archive directory без полной распаковки на диск.
2. Извлекать только text-like members и archive-like members.
3. Прогонять members через тот же fast classifier.
4. Если member начинается с ZIP signature, пытаться смотреть его contents рекурсивно.
5. Бинарные non-text members считать отдельно как skipped.

### 3. Что делать с `.txt`, которые на деле ZIP

Если top-level `.txt` начинается с `PK\\x03\\x04`, его не надо считать plain text.
Его надо открывать как ZIP и оценивать contents по тем же правилам.
